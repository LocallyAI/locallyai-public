# One-click start & stop

When to read: any time you want to run LocallyAI without opening a
terminal. After install, this is the supported start/stop path for
end users and IT-ops.

For first-time install (Python venv, ollama, TLS certs, .env), see the
relevant setup chapter — [setup-mac-single.md](setup-mac-single.md),
[setup-mac-ha.md](setup-mac-ha.md), or [setup-windows.md](setup-windows.md).
This chapter assumes install completed successfully at least once.

---

## What it starts

There are two launcher apps — one per UI — so end users and admins
can start only what they need.

| App | Starts | Opens |
|---|---|---|
| **LocallyAI Worker.app** | API + worker-ui | Worker UI tab (5174) |
| **LocallyAI Manager.app** | API + manager-ui | Manager UI tab (5173) |
| **Stop LocallyAI.app** | nothing | nothing — boots out everything |

The API is shared. Click both apps and you'll have all three services
running; click only one and only that UI's dev server runs (saves
~150 MB of vite dev memory if you only use one side).

### Warm vs cold

Each launcher uses macOS `launchd` to keep the services it cares
about warm between clicks:

| Click # | What happens | Time on a Mac Studio |
|---|---|---|
| 1 — first ever | Writes the relevant LaunchAgents, bootstraps them, waits for ports, opens the tab | ~1.5–2 s |
| 2 — services already running | Just opens the browser tab | ~200–300 ms |
| n+1 after stop | Fresh install + bootstrap (same as click 1) | ~1.5–2 s |

So once you've started a UI, **every subsequent click of its app is
effectively instant** — services don't re-spin-up between clicks.
They restart automatically on crash (KeepAlive=true).

The Stop app is the only thing that takes them down: it `bootout`s
all three labels (API, worker-ui, manager-ui) and **deletes the
plists** so they don't auto-load on your next login. Re-launch
either start app any time.

## Mac (recommended)

### First time

1. In Finder, navigate to the LocallyAI folder.
2. Double-click **LocallyAI Worker.app** (or **LocallyAI Manager.app**).
3. macOS will ask "is this app safe to open?" the first time —
   right-click the app, choose **Open**, then click **Open** again
   in the dialog. This grants permission to launch unsigned apps;
   only required once per app (so you'll do it once for Worker and
   once for Manager).
4. **One-time HTTPS cert acceptance** (HTTPS deployments only): if
   the UI loads but shows "cannot connect to server" the very first
   time, visit `https://localhost:8000/healthz` in the same browser
   and click **Advanced ▶ Proceed to localhost (unsafe)**. From then
   on the browser silently accepts the self-signed cert and the UI
   tab works. (The launcher no longer auto-opens the healthz tab —
   we found the extra browser flicker more annoying than the rare
   first-run prompt it solved.)
5. The relevant UI tab opens:
   - **Worker** → http://localhost:5174/ (paste your user API key)
   - **Manager** → http://localhost:5173/ (paste the admin API key)
6. A macOS notification confirms which one is running.

> **HTTP fallback.** If you don't have TLS certs (no `tls/key.pem`)
> or you set `LOCALLYAI_ALLOW_HTTP=1` in `.env`, the launcher starts
> the API as plain HTTP. In that case both UIs' `.env.local` must
> use `http://localhost:8000` (worker-ui ships with `https://...`
> so update both files to match).

### Every other time

Double-click whichever launcher you need. Tab opens. Done.

If the services it relies on are already running, the launcher
detects that and skips straight to the browser-tab open — no
double-start, no port conflict.

### Stopping

Double-click **Stop LocallyAI.app**. All three processes are
SIGTERMed (with a 2-second grace) and the PID file is cleared. A
notification confirms.

You can also leave them running indefinitely — they're light
(uvicorn idle is ~80 MB; vite dev is ~150 MB each). The OS reclaims
memory if you sleep / reboot.

### Where everything lives

| Where | What |
|---|---|
| `LocallyAI Worker.app` | Launcher: starts API + worker-ui, opens worker tab |
| `LocallyAI Manager.app` | Launcher: starts API + manager-ui, opens manager tab |
| `Stop LocallyAI.app` | Stopper: kills all three services + removes their plists |
| `scripts/start_locallyai.sh [worker\|manager\|both]` | The actual launcher logic |
| `scripts/stop_locallyai.sh` | The actual stop logic |
| `~/Library/LaunchAgents/app.locallyai.{api,worker-ui,manager-ui}.plist` | The 3 LaunchAgents that keep services warm. Generated by start, deleted by stop. |
| `logs/launcher/launcher.log` | Launcher's own log (start, stop, errors) |
| `logs/launcher/api.log` | API server stdout + stderr |
| `logs/launcher/worker-ui.log` | Worker UI dev server output |
| `logs/launcher/manager-ui.log` | Manager UI dev server output |
| `logs/launcher/.backend` | `launchd` or `nohup` — which path the last start used |

### Moving the .app to /Applications or the Dock

The .app expects to live next to `api.py` (in the repo root) by
default. If you'd like it in `/Applications`, the Dock, or anywhere
else:

```bash
# Tell the .app where the repo is.
mkdir -p ~/Library/Application\ Support/LocallyAI
echo "$(cd /path/to/locallyai && pwd)" > ~/Library/Application\ Support/LocallyAI/repo_path

# Then drag the .app to /Applications (or anywhere). It reads the path
# above when it can't find api.py next to itself.
```

Both apps (LocallyAI.app and Stop LocallyAI.app) read the same
`repo_path` file.

### Auto-start at login

The launcher uses LaunchAgents (`~/Library/LaunchAgents/app.locallyai.*`),
**but** the stop app deletes those plists. So out of the box, after a
reboot or logout, services are NOT running until you click the start
app once.

If you'd rather have LocallyAI come up automatically on every login
without any click:

1. **System Settings → General → Login Items**.
2. Click **+**, navigate to **LocallyAI.app**, click **Open**.
3. (Optional) Tick the "Hide" checkbox so no app icon flashes.

The launcher is idempotent — on a fast login where services are
already running, it just opens the browser tabs in ~250 ms.

**Power users** who want services to survive reboots without going
through the .app, copy the plists out of `~/Library/LaunchAgents/`
to a backup location *after* a successful start, and copy them back
after a stop. The agents auto-load when present at login.

### Manual launchd inspection

```bash
# What's running?
launchctl print "gui/$(id -u)/app.locallyai.api"
launchctl print "gui/$(id -u)/app.locallyai.worker-ui"
launchctl print "gui/$(id -u)/app.locallyai.manager-ui"

# Force-restart one service (without taking down the others):
launchctl kickstart -k "gui/$(id -u)/app.locallyai.worker-ui"
```

## Windows

### First time

1. In File Explorer, navigate to the LocallyAI folder.
2. Double-click **start_locallyai.vbs** (in the repo root).
3. Windows SmartScreen may warn "Windows protected your PC" — click
   **More info** → **Run anyway**. (Once.)
4. Two browser tabs open:
   - http://localhost:5174/ — Worker UI
   - http://localhost:5173/ — Manager UI

### Every other time

Double-click **start_locallyai.vbs**. Done.

### Stopping

Double-click **stop_locallyai.vbs**.

### Auto-start at login

1. Press **Win+R**, type `shell:startup`, hit Enter.
2. Drop a shortcut to `start_locallyai.vbs` into that folder.

### Where everything lives

| Where | What |
|---|---|
| `start_worker.vbs`  | Launcher: starts API + worker-ui, opens worker tab |
| `start_manager.vbs` | Launcher: starts API + manager-ui, opens manager tab |
| `stop_locallyai.vbs` | Stopper |
| `scripts\start_locallyai.bat [worker\|manager\|both]` | The .bat the .vbs's run hidden |
| `scripts\stop_locallyai.bat`  | The stop .bat |
| `logs\launcher\*.log` | Same files, same purpose as Mac |

## Troubleshooting

### "Nothing happens when I double-click"

Tail the launcher log:
```bash
tail -50 logs/launcher/launcher.log
```
Common causes (the script writes a clear `ERROR:` line for each):
- **Python venv missing** — `bash install.sh` to recreate it.
- **UI deps missing** — `cd apps/worker-ui && npm install`, same for manager-ui.

### "Browser tabs open but show 'cannot connect to server'"

Most common cause is a **TLS scheme mismatch**: the API was launched
as HTTP but the UI's `.env.local` says `https://localhost:8000` (or
the reverse). Both must match.

Check what's actually running:
```bash
ps -ef | grep uvicorn | grep -v grep
# Look for --ssl-keyfile / --ssl-certfile flags. If present → HTTPS.
```
Check what each UI expects:
```bash
grep VITE_API_BASE_URL apps/worker-ui/.env.local apps/manager-ui/.env.local
```
Both UIs and the running API must agree (https or http). After fixing
`.env.local`, restart vite dev (Stop LocallyAI.app + LocallyAI.app)
so it re-reads the file.

The dev servers can also be slow to bind on a cold start (5–10 s) —
refresh once or twice. If still failing, tail the per-service log:
```bash
tail -50 logs/launcher/api.log
tail -50 logs/launcher/worker-ui.log
tail -50 logs/launcher/manager-ui.log
```

### "Manager UI never loads but worker UI works (or vice versa)"

CORS allowlist mismatch. The API only accepts cross-origin requests
from origins listed in `LOCALLYAI_CORS_ORIGINS` in `.env`. Both UI
ports (5173 + 5174, in both `localhost` and `127.0.0.1` form) must
be present:
```
LOCALLYAI_CORS_ORIGINS=http://localhost,http://127.0.0.1,http://localhost:5174,http://127.0.0.1:5174,http://localhost:5173,http://127.0.0.1:5173
```
Restart the API after editing.

### "Port already in use"

Something else is on 8000/5173/5174. Either:
- Stop the conflicting process (typically a stray vite from a
  previous dev session — `bash scripts/stop_locallyai.sh` clears
  these).
- Override the port with `LOCALLYAI_API_PORT`, `LOCALLYAI_WORKER_PORT`,
  or `LOCALLYAI_MANAGER_PORT` env vars (set in `.env`, then
  re-launch).

### "Stop doesn't actually stop"

The stopper takes two passes (SIGTERM then SIGKILL after 2 s) and
also sweeps each port for stragglers. If it's still failing:
```bash
bash scripts/stop_locallyai.sh   # try again with verbose output in launcher.log
ps -ef | grep -E "uvicorn|vite|npm" | grep -v grep
```
Kill anything stuck by hand: `kill -9 <pid>`.

### "I want to see what's happening live"

Run the script directly from a terminal — same logic, just visible:
```bash
bash scripts/start_locallyai.sh
tail -F logs/launcher/api.log logs/launcher/worker-ui.log logs/launcher/manager-ui.log
```
