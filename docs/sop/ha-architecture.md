# HA architecture — two-Mac active/standby pair

**Every LocallyAI install ships with two Mac Studios.** This chapter
is the architecture reference: what gets synchronised between them,
how failover happens, how the vendor monitors it. The setup procedure
itself lives in [setup-mac-ha.md](setup-mac-ha.md); the
failover-readiness drill the firm runs every quarter lives in
[`../runbooks/failover-test.md`](../runbooks/failover-test.md).

## Topology

```
                         Office LAN (single subnet)
                         ┌──────────────────────────────────┐
                         │                                  │
              ┌──────────┴───────────┐         ┌────────────┴─────────┐
              │  Mac-A (PRIMARY)     │         │  Mac-B (STANDBY)     │
              │  192.168.x.10        │◄ rsync ►│  192.168.x.11        │
              │  office-a.local      │ Synct.  │  office-b.local      │
              │                      │         │                      │
              │  - API :8000 (TLS)   │         │  - API :8000 (TLS)   │
              │  - manager-ui :5173  │         │  - manager-ui :5173  │
              │  - worker-ui :5174   │         │  - worker-ui :5174  │
              │  - Qdrant (replica)  │◄ HTTP ► │  - Qdrant (replica)  │
              │  - audit.log (own)   │         │  - audit.log (own)   │
              │  - SHARED_DIR ───────┴── shared volume ───────────────┤
              └──────────────────────┘         └──────────────────────┘
                         ▲                                ▲
                         │                                │
                         └─────── Staff laptops ──────────┘
                              (worker-ui smart client
                               retries primary→standby)
```

## What syncs between the two Macs

Three categories of state, three different sync mechanisms.

### 1. Shared identity + governance state — Syncthing

Lives at `SHARED_DIR/` (set via `LOCALLYAI_SHARED_DIR`):

| File | Purpose | Why shared |
|---|---|---|
| `users.json` | API keys (pseudonymised at hash) | Key issued on Mac-A must work on Mac-B |
| `erasure.log` | GDPR Art. 17 tombstones | Erasure on Mac-A must block writes on Mac-B |
| `fleet.json` | Live node registry | Smart client uses this to discover peers |
| `doc_acls.json` | Per-doc ACLs (per `document-acl.md`) | Restriction on Mac-A must apply on Mac-B |

**Tool**: [Syncthing](https://syncthing.net) — peer-to-peer file sync,
no cloud. Already wrapped by vendor-shipped install scripts.
**Cadence**: real-time (Syncthing watches inotify-equivalents).
**Conflict policy**: documented in `sync_conflicts.py` — file-level
conflicts are quarantined to `SHARED_DIR/conflicts/` and the breach
detector raises a CRITICAL alert (no silent merge of credential or
erasure data).

### 2. Document corpus — rsync (primary → standby)

`data/uploads/` (and the embedded Qdrant index, when not using a
Qdrant server cluster) is much larger and changes less frequently.
For these:

**Recommended**: an `rsync` job from primary → standby, scheduled
hourly via launchd. The job is **operator-triggered first run**
(after install or after a bulk-ingest), then runs hourly thereafter.

The launchd plist `app.locallyai.ha-rsync.plist` is generated by
`install.sh` when `LOCALLYAI_HA=1` is set. It runs:

```bash
rsync -avh --delete \
  --exclude='*.tmp' --exclude='.lock' \
  primary-host:~/locallyai/data/uploads/ ~/locallyai/data/uploads/

rsync -avh --delete \
  primary-host:~/locallyai/storage/ ~/locallyai/storage/
```

**Cadence**: hourly. Sufficient for a research corpus (lawyers don't
expect their just-uploaded doc to be retrievable on the standby in
under an hour).

**Why not Syncthing for `data/uploads/`?** Syncthing is bidirectional
by default; for a large corpus that's wasted disk I/O on the standby
side. rsync one-way primary→standby is cleaner.

**Why not just Qdrant clustering for vectors?** Qdrant clustering is
supported (set `QDRANT_URLS=http://primary:6333,http://standby:6333`
+ `LOCALLYAI_HA=1` — the `ensure_collection` code already configures
`shard_number=2 replication_factor=2`). It's the right answer for
high-write firms. For the typical install where ingest is bursty +
read-heavy, embedded Qdrant + rsync is simpler.

### 3. Real-time per-node state — NOT synced

Each Mac maintains its own:

- `audit.log` — per-node HMAC chain. The chain head is per-node;
  cross-node audit verification happens via `/admin/fleet/audit-verify`
  which checks all nodes' chains independently.
- `billing.log` — same model.
- `security.log` — per-node breach detector state.
- The current API process state, in-flight requests, KV cache.

This is intentional — synchronising per-node logs would mean the
chain has to handle cross-writer interleaving. Per-node chains let
each Mac fail independently without corrupting evidence.

## Failover model

**Active-standby with smart-client retry.** Both Macs run the full
stack at all times. Lawyers' worker-ui apps know about both URLs and
fail over automatically.

### How a staff laptop reaches the office Macs

The Workspace.app (and Manager.app) is built per-firm with a baked
URL. For HA installs, build with the **load-balancer hostname** the
firm's IT sets up, OR use the smart-client multi-URL config:

```bash
# Per-firm build — set both hostnames at build time
WORKSPACE_URL=http://office-a.local:5174 ./build.sh
# AND set VITE_API_BASE_URLS in apps/worker-ui/.env to include both:
VITE_API_BASE_URLS=https://office-a.local:8000,https://office-b.local:8000
```

The smart client (`apps/worker-ui/src/lib/api.ts`) iterates the URL
list with mtime-cached health checks every 5 seconds:

1. Try primary URL → if 200, use it
2. Primary fails → try standby
3. In-flight POSTs carry a `client_request_id` so the server-side
   dedup cache prevents double-billing if the request actually
   completed on the primary before the connection dropped

### When the primary recovers

The standby continues serving until the staff laptop's next health
check (5 s default) detects primary is healthy again. **The standby
does not "yield"** — it stays primary-of-record until the next quiet
moment when the operator manually re-promotes Mac-A. Avoids
flapping during transient network blips.

### What the vendor monitors (Cloudflare worker)

Per the data-isolation chapter, both Macs send heartbeats every 5
minutes carrying `firm_id` (same hash for both — same firm) and
`node_id` (different per Mac). The vendor monitor dashboard groups
heartbeats by `firm_id` and renders both nodes side-by-side under
the firm's card.

Alerts the dashboard fires:

| Condition | Severity | Action |
|---|---|---|
| One node silent > 10 min | warning | Operator notified; check the silent Mac |
| Both nodes silent > 10 min | critical | Wake-up call to firm IT — full outage |
| Nodes disagree on `users.json` count | warning | Syncthing lag or split-brain; investigate |
| `fleet.json` shows `active_nodes != 2` | warning | One node didn't register on startup; check launchd |
| `audit_chain` differs across nodes (per-node verify) | critical | Tamper or drift — `audit-chain-broken.md` runbook |

## What HA does NOT solve

| Problem | Mitigation |
|---|---|
| Power outage at office | Both Macs go down. UPS recommended for the rack |
| Office LAN failure | Tailscale on both Macs as a fallback path for staff laptops |
| Whole-office disaster (fire, flood) | Out of scope — DR per `docs/sop/recovery.md` (off-site backup of `data/uploads/` to S3-class storage with FileVault keys) |
| Both Macs running the same buggy release | Update one Mac at a time; smoke for 24 h; then update the second (rolling update per `maintenance.md`) |
| Quadrant cluster split-brain | `write_consistency_factor=2` makes the cluster read-only on partition rather than allowing silent divergence |

## Files this depends on

- `install.sh` — generates the rsync launchd plist when `LOCALLYAI_HA=1`
- `fleet.py` — node registration in `fleet.json`
- `sync_conflicts.py` — Syncthing conflict detection + quarantine
- `apps/worker-ui/src/lib/api.ts` — smart client multi-URL retry
- `apps/manager-ui/src/lib/api.ts` — same multi-URL pattern (Manager UI usually points at one node, but smart-client retry works the same)
- `setup-mac-ha.md` — operator setup procedure (Syncthing pair, clock sync, network checks)
- `../runbooks/failover-test.md` — quarterly drill
- Vendor monitor dashboard `Fleet` tab — per-firm grouping with both nodes' status
